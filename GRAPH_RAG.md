# Graph RAG in Scout

> Design note for the schema expansion (34 tables) + Graph RAG implementation.

## 1. What "Graph RAG" means here

Classic RAG retrieves relevant **documents** and stuffs them into the prompt. **Graph
RAG** retrieves a relevant **subgraph** of a knowledge graph instead, so the model
gets not just the nodes but the *relationships* between them.

Scout's data is a structured ClickHouse warehouse, not a document corpus, so the
knowledge graph is the **schema graph**:

- **Nodes** = tables (34 of them).
- **Edges** = the join keys that link them.

The catch: ClickHouse has **no foreign keys**, and the warehouse is deliberately built
that way (the brief was "30+ interconnected tables with no explicit foreign key"). The
relationships are *implicit* - `disputes.txn_id` lines up with `card_transactions.txn_id`,
`collections.assigned_employee_id` lines up with `employees.employee_id` - but nothing in
the database declares them. **Graph RAG's job is to recover that graph and use it to
retrieve the right tables + join paths for each question.**

### Why this was needed

The old agent fed the planner a **flat catalog** ("here are all the tables, pick some").
At 6 tables that is fine. At 34 tables a single business question routinely spans a
join chain the planner can't see - *"which high-value customers had a disputed card
transaction, and which branch issued their account?"* needs
`customers → cards → accounts → branches` **and** `card_transactions → disputes`. With no
FKs and a flat list, the model guesses join keys and gets them wrong. The schema graph
makes those paths explicit.

## 2. The expanded schema (34 tables)

The warehouse grew from 6 to 34 tables, all in one coherent card-issuer / retail-bank
domain and all linked only by shared key columns (no FKs):

| Sub-domain | Tables |
|---|---|
| Customer & geography | `customers`*, `geographies`, `devices` |
| Branch & staff | `branches`, `employees` |
| Accounts & cards | `accounts`, `cards`, `card_products`, `card_applications`, `account_transactions` |
| Payments & rewards | `card_transactions`*, `statements`, `rewards_ledger`, `reward_redemptions`, `offers`, `offer_redemptions` |
| Lending | `loan_book`*, `loan_products`, `loan_applications`, `loan_repayments`, `collections`, `credit_bureau` |
| Risk & compliance | `disputes`, `fraud_alerts`, `kyc_records`, `aml_screenings` |
| Merchants | `merchants`, `merchant_categories` |
| Engagement & marketing | `app_sessions`, `support_tickets`, `marketing_campaigns`*, `campaign_responses` |
| Retail (pre-existing) | `market_sales`*, `ecommerce_orders`* |

`*` = the 6 pre-existing tables. The 28 new ones are generated by
[`scripts/seed_graph.mjs`](scripts/seed_graph.mjs).

### The data is derived, not random

A graph of tables is only useful if the rows actually reconcile across them. The seeder
enforces that two ways:

1. **Derived from real facts** - `rewards_ledger` and `statements` are computed from the
   actual `card_transactions` (points and monthly spend reconcile *exactly*);
   `disputes`/`fraud_alerts` are drawn from transactions genuinely flagged `is_fraud`;
   `loan_repayments` is the real EMI schedule of each loan; `collections` exist **only**
   for delinquent/NPA/written-off loans.
2. **One per-customer profile** seeded from each customer's existing
   `value_band`/`credit_score`/`income`/`status` drives every related row, so card
   limits track income, bureau scores track credit scores (corr ≈ 0.99), and dormant
   customers show sparse engagement.

The seeder ends with a **reconciliation pass** (12 checks) that proves it:

```
✅ cards / accounts / statements / loan_repayments / disputes → parents (orphans = 0)
✅ collections ONLY on delinquent loans
✅ rewards_ledger earn = card_transactions reward points (exact)
✅ statements spent ≈ card_transactions spent (<0.1% gap)
✅ kyc_records.status matches customers.kyc_status
✅ credit_bureau scores track customer credit (corr 0.99)
12/12 reconciliation checks passed.
```

## 3. The Graph RAG layer (`lib/graph/`)

### `relationships.ts` - recovering the edges

Two complementary sources, merged (curated wins on conflict):

- **Auto-inference**: any key-like column (`*_id`, plus known join columns) that a table
  shares with its *canonical parent* table becomes an edge. This finds the bulk of the
  graph with **zero FK metadata** - the point being that the relationships are
  discoverable even though the database never declares them.
- **Curated manifest**: the aliased edges inference can't see by name -
  `collections.assigned_employee_id → employees.employee_id`,
  `card_transactions.merchant → merchants.merchant_name`,
  `loan_book.branch → branches.branch_id`, etc.

`customer_id` and `city` are tagged as **hub columns** and traversal penalises them, so a
question about `disputes` doesn't drag the whole warehouse in through `customers`.

### `schema-graph.ts` - retrieval

- `buildSchemaGraph(catalog)` - nodes from the cached catalog, edges from the above,
  cached and rebuilt only when the catalog changes.
- `retrieveSubgraph(graph, seedTables)` - the heart of it: starting from the planner's
  seed tables, (1) keep the seeds, (2) add the bridge tables on the shortest **join
  path** between them, (3) enrich with their strongest direct neighbours up to a cap.
- `findJoinPath(graph, a, b)` - Dijkstra over edge weights → the concrete join chain.
- `formatGraphForPrompt(subgraph)` - renders the subgraph as a `JOIN GRAPH` block of
  `tableA.colA = tableB.colB` edges that the analyst LLM reads.

You can see all of this with **no LLM**:

```bash
npm run db:graph                       # full edge list + demo subgraphs
npm run db:graph -- disputes branches  # the subgraph for a seed set
```

## 4. How the agent uses it

The pipeline went from 5 phases to 6 - a **RELATE** phase slots in between PLAN and
INSPECT ([`lib/agent/workflow.ts`](lib/agent/workflow.ts)):

```
DISCOVER → PLAN → RELATE → INSPECT → ANALYZE↻ → SYNTHESIZE
                  ▲ Graph RAG: planner's seed tables → connected subgraph + join keys
```

1. **PLAN** picks a few seed tables from the question.
2. **RELATE** ([`phases.ts:relate`](lib/agent/phases.ts)) walks the schema graph from those
   seeds, expands to the connected subgraph (pulling in bridge tables like `accounts`),
   and emits a live **"Mapped table relationships"** step chip
   (e.g. *"linked 6 tables via customer_id, card_id, branch_id (+2 bridge)"*).
3. **INSPECT** describes the subgraph's tables (now up to 8, not 4).
4. **ANALYZE** gets the `JOIN GRAPH` block in its prompt and is instructed to join *only*
   on those recovered keys, chaining through bridge tables to reach far tables.

The change is **additive and safe**: if the graph is empty or unreachable, RELATE falls
back to the seed tables and the pipeline behaves exactly as before.

## 5. Try it end-to-end

```bash
npm run dev   # needs OPENAI_API_KEY + ClickHouse creds in .env
```

A question that is impossible without the graph:

> "Which high-value customers had a disputed card transaction in the last year, and
> which branch issued their account?"

Watch the **Mapped table relationships** chip light up, then the analyst join
`customers → cards → accounts → branches` and `card_transactions → disputes` on the exact
keys the graph handed it.
